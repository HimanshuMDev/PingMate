# PingMate – Voice AI Analysis & Local AI Recommendations

## 1. Current Flow (What You Have)

| Step | Where | What Happens |
|------|--------|--------------|
| 1 | **HomeScreen.kt** | User taps the blue FAB (AI Assistant). If `RECORD_AUDIO` is granted → `VoiceAiDialog` opens and `viewModel.startVoiceAi()` runs. |
| 2 | **SpeechRecognizerManager.kt** | Android `SpeechRecognizer` captures speech. Partial + final results → `transcription` StateFlow. When user stops speaking, `isListening` becomes false. |
| 3 | **HomeScreen.kt** (LaunchedEffect) | When `!isListening` and `transcribedText` is not blank and dialog is visible → calls `viewModel.processAiPrompt(transcribedText)`. |
| 4 | **HomeViewModel.kt** | `processAiPrompt(prompt)` sets "Analyzing Prompt...", then on IO calls `summarizationEngine.summarize(prompt)` and shows the result in `aiResponse`. |
| 5 | **OfflineSummarizationEngine.kt** | **No LLM.** Uses: (a) keyword map for app (whatsapp, gmail, etc.), (b) ML Kit Entity Extraction for person names and dates, (c) brute-force token/substring match on notification **titles** for sender, (d) rule-based summary (questions / actions / chatter) + ML Kit Smart Reply. |

**Data source:** Only what **PingMateNotificationService** stores from **status bar notifications** (title, content, timestamp, packageName). No access to SMS DB or WhatsApp/other in-app chat history.

---

## 2. Why Results Feel “Random” or Wrong

1. **No real understanding of the prompt**  
   The app never interprets *what you’re asking*. It only:
   - Detects app keywords (“whatsapp”, “today”, etc.)
   - Extracts person/date via ML Kit (often misses informal or non-English names)
   - Matches sender by **first** token/substring match on **notification title**  
   So “summarize messages from John” can match the wrong “John” or no one.

2. **Fragile sender resolution**  
   In `OfflineSummarizationEngine`:
   - If ML Kit doesn’t return a person, it uses `promptTokens` and checks if any **title** contains a token (or the reverse). First match wins.
   - So “what did Dave send” might match “David” or “Dave’s Pizza” or another app’s notification with “Dave” in the title.

3. **Template-only “summary”**  
   The “summary” is not generated by a model. It’s fixed logic:
   - Notifications with `?` → “questions”
   - Notifications containing “please”, “can you”, “urgent”, etc. → “actions”
   - Rest → “chatter”  
   So you can’t get a true “one-sentence summary” or “what did X ask me?” — only this canned structure.

4. **Data limits**  
   Only notification content is available (what apps put in the notification), not full threads. So “today’s chat” is really “today’s notifications we stored for the selected app/sender.”

5. **No question-answering**  
   There is no component that takes a natural-language question and answers from the stored notifications. So “any question” cannot be answered correctly — only the hard-coded summary + Smart Reply.

---

## 3. What You Want (Target Behavior)

- **Local AI** (on-device, no mandatory cloud).
- **Understand** the spoken prompt (e.g. “summarize my today’s chat”, “what did [person] send me”).
- **Summarize** what you said and **analyze** it to produce a **proper summary** of the relevant chats/notifications.
- Support **arbitrary questions** over your chats and get **correct, relevant answers**.

To support “any question” and proper summarization, you need something that can **read and reason over text** — i.e. a **local LLM** (or a cloud LLM with strict privacy, if you later choose that). Rule-based + ML Kit alone cannot do this reliably.

## 4. Recommended Directions for Local AI

### Option A: **Gemini Nano (AICore) – Android 15+**

- **What:** Google’s on-device LLM via **AICore** (Gemini Nano). Runs on supported devices (e.g. Pixel 9, and expanding).
- **Pros:**  
  - First-class Android API, good for “summarize this” / “answer from this context”.  
  - You pass your **notification text as context** and the **user prompt** (transcribed voice), and get a real summary or answer.  
  - No backend to maintain; on-device and private.
- **Cons:**  
  - Requires Android 15+ and a device that supports AICore.  
  - Not all devices have it yet.
- **Use case:** “Summarize my today’s WhatsApp messages”, “What did John ask me?” → you build a context string from `NotificationDao` (filtered by app/sender/date like now) and send it + the user prompt to Gemini Nano; it returns the summary/answer.

**High-level flow:**  
`Voice → STT → transcribed prompt` + `notification context string` → **Gemini Nano** → display model response in `VoiceAiDialog`.

---

### Option B: **MLC LLM / Llama.cpp on Android**

- **What:** Run small open-source LLMs (e.g. **SmolLM**, **TinyLlama**, **Phi-2**) **on device** via MLC LLM Android or llama.cpp Android bindings.
- **Pros:**  
  - Fully offline, no dependency on Google or any cloud.  
  - Works on any device that can run the model (memory/CPU dependent).  
  - You control the exact prompt: “Here are my recent notifications: … User question: …”
- **Cons:**  
  - Need to integrate a native/lib layer and manage model files.  
  - Smaller models may be less accurate than Gemini Nano; need to keep context (notification text) short to fit context window.
- **Use case:** Same as above: build context from notifications + transcribed prompt → one prompt to the local LLM → show response.

---

### Option C: **Improve Current Pipeline (No LLM)**

- **What:** Keep ML Kit + rules but improve **intent detection** and **sender/app/date filtering** so the right notifications are selected and the template summary is at least about the **correct** person and time.
- **Pros:** No new dependencies, works on all devices.  
- **Cons:** Still no “ask any question”; only better structured summaries (e.g. “Questions from X today”, “Actions from Y”). Not true “analyze and answer anything.”

**Improvements you can do now:**
- Use **full name / multi-token** matching for sender (e.g. prefer “John Doe” over “John” when both exist).
- Add explicit intents: e.g. “summarize today” vs “what did [X] send” vs “questions from [X]” and format the summary accordingly.
- Use **recency and frequency** (e.g. most notifications from that sender today) to disambiguate when multiple senders match.

This reduces “random” results but does **not** give you a general “ask any question and get a proper result” experience.

## 5. Suggested Path for “Analyze Chats + Answer Any Question”

1. **Short term (no new SDK):**  
   - Implement **Option C** so that sender/app/date filtering and intent handling are more robust.  
   - Show the **filtered notification list** (or a clear “Context” section) in the Voice AI dialog so the user sees what was used for the “summary”.  
   - This makes the current “summary” at least **consistent and explainable**.

2. **Medium term (local AI that can really answer):**  
   - Prefer **Option A (Gemini Nano)** if your target devices are Android 15+ and AICore is available: least integration effort for a real “ask anything over my notifications” experience.  
   - If you need **maximum compatibility** or **fully vendor-independent** offline: use **Option B** (MLC LLM or llama.cpp) with a small model and a prompt that includes “notification context + user question”.

3. **Data side:**  
   - Keep in mind: the only “chats” you have are **notification-derived** (title, content, time, app). For “complete chats” you’d need either:
     - Read access to SMS/MMS (if you add that and the user grants it), or  
     - Other app-specific APIs (e.g. backup/export), which are often restricted.  
   So “analyze my complete chats” today means “analyze all the notification-based messages we have for the selected app/person/date.”

## 7. Best choice: 100% free, offline, all devices, shareable APK

For your requirements:

- **Completely free** – no paid APIs or subscriptions  
- **Works on all devices and Android versions** – no Android 15 or specific hardware  
- **Offline** – no internet after setup  
- **Share APK** – someone installs your APK and can use the app; no PC or “local system”  
- **Good accuracy** – real LLM that can summarize and answer questions  
- **Reasonable speed** – small model runs on CPU on most phones  

the recommended approach is:

### Use **Llamatik** + a small **GGUF model** (first-run download)

| What | Details |
|------|--------|
| **Library** | **Llamatik** (`com.llamatik:library`) – Kotlin, runs on Android (min SDK 26), uses llama.cpp under the hood. Free, open-source, no vendor lock-in. |
| **Model** | A small instruction-tuned GGUF model, e.g. **SmolLM-360M-Instruct** or **TinyLlama-1.1B** in **Q4** quantization. ~50–270 MB depending on model. |
| **How it’s distributed** | **First-run download**: APK stays small (~15–25 MB). On first launch, the app downloads the model once (from your server, GitHub Releases, or Hugging Face) into app storage. After that, everything runs **fully offline**. When you share the APK, the other person installs it, opens the app, and the first time they see “Downloading model…” (one-time, needs internet). No extra app, no PC, no payment. |
| **Accuracy** | Real LLM: understands “summarize today’s chat”, “what did John send me?”, “any question” over your notification context. Much better than rule-based logic. |
| **Speed** | Small models (360M–1.1B params) run on CPU in a few seconds on typical phones. Optional: use a smaller/faster model if you prefer speed over quality. |
| **Flexibility** | No dependency on Android version or device brand. Works on API 26+ (your app is 29+). |

### Why this fits “share APK and use directly”

- You share **one APK**. The recipient installs it and opens the app.  
- **First open**: app checks if the model is already in app storage. If not, it shows a one-time “Download model (~50–150 MB)” and downloads it (needs Wi‑Fi/mobile data once).  
- **After that**: model is stored on device; all summarization and Q&A run **offline**. No “local system” or extra setup.  
- **Alternative (zero download)**: If you want “install and use with no download at all”, you’d have to bundle a very small model inside the APK. That would require a model under ~80 MB to stay under the 100 MB APK limit and would mean lower quality. First-run download is the better trade-off: small APK, one-time download, then full offline + good accuracy.

### Integration outline

1. Add dependency: `implementation("com.llamatik:library:0.16.0")` (check [Llamatik](https://github.com/ferranpons/Llamatik) for latest).  
2. **Model**: Host the GGUF file (e.g. SmolLM-360M-Instruct Q4) somewhere free (e.g. GitHub Releases, or Hugging Face + your app downloads via URL). On first launch, if the model file is not in `getFilesDir()`, download it and save there.  
3. **Load model once** (e.g. in `Application` or when Voice AI is first used): `LlamaBridge.getModelPath("your-model.q4.gguf")` if bundled, or use the path where you downloaded the file; then `LlamaBridge.initGenerateModel(modelPath)`.  
4. **Replace current “summarization”**: Build a **context string** from your notifications (same filtering you have now: app, sender, today, etc.). Call `LlamaBridge.generateWithContext(systemPrompt, contextBlock, userPrompt)` where `contextBlock` is the notification text and `userPrompt` is the transcribed voice. Show the returned string in `VoiceAiDialog`.  
5. **System prompt** example: *“You are an assistant. Given the following list of notifications/messages, answer the user’s question concisely. If the user asks for a summary, summarize. If they ask what someone said, answer based only on the context.”*

### Summary

- **Best suggestion**: **Llamatik** + **SmolLM-360M-Instruct** (or similar small GGUF) with **first-run in-app model download**.  
- **Free, offline (after first run), works on all devices/versions, shareable APK, no extra setup for the user, and real accuracy** for summarizing and answering questions over your notifications.
